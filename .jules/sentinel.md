## 2026-06-01 - [Fix XSS in status_dashboard.html]
**Vulnerability:** A Cross-Site Scripting (XSS) vulnerability existed in `frontend/templates/status_dashboard.html` where untrusted configuration settings (`value`), external service messages (`data.message`), and token expirations (`data.token_info.expires_in_human`) were injected directly into the DOM via `.innerHTML` without sanitization.
**Learning:** Even internal or admin-focused dashboards can be vulnerable if they display external or user-configurable data without escaping. Constructing HTML strings dynamically from unvalidated sources is a common vector for DOM-based XSS.
**Prevention:** Always use a sanitization function like `escapeHtml` to escape dangerous characters (`<`, `>`, `&`, `"`, `'`) before assigning dynamic content to `.innerHTML`, or prefer `.textContent` when only plaintext is intended.
## 2026-06-21 - [Fix XSS in frontend error messages]
**Vulnerability:** A DOM-based Cross-Site Scripting (XSS) vulnerability existed where error messages from API responses (like `err.message` in `file_view.html` and `files.html`) were injected directly into `.innerHTML` without sanitization. An attacker could potentially return malicious error messages from the backend or proxy to execute arbitrary JavaScript in the user's browser. Additionally, the `escapeHtml` utility function was defined redundantly across multiple HTML files instead of being centralized.
**Learning:** External or user-controlled error messages must always be sanitized before being displayed via `.innerHTML`. Furthermore, duplicating security utility functions across multiple files leads to inconsistent application and maintenance overhead.
**Prevention:** Centralize security utilities like `escapeHtml` in a common JS file (e.g., `frontend/static/js/common.js`) and expose them globally. Ensure that all dynamic content assigned to `.innerHTML` uses this centralized sanitization function.
